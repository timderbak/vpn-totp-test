# CLAUDE.md

Правила работы Claude Code в репозиториях, созданных из этого шаблона. Файл оптимизирован под **соло-вайбкодера**.

База — четыре принципа Karpathy против типичных болезней LLM-кода (см. [оригинал](https://x.com/karpathy/status/2015883857489522876) и [интерпретацию](https://github.com/forrestchang/andrej-karpathy-skills)) + локальные практики: UI-первичность, Docker, тесты, research-first, секреты, контекст.

> Дисциплина задачи держится на плагине **superpowers** (`brainstorming`, `writing-plans`, `test-driven-development`, `systematic-debugging`, `verification-before-completion`).
> Для многофазных проектов — опционально **GSD** (`.planning/PROJECT.md`, фазы). См. [`docs/gsd-vs-superpowers.md`](docs/gsd-vs-superpowers.md).

- Код, имена, docstrings, git-коммиты, PR-описания, логи, комментарии в коде — на английском.

## 1. Четыре принципа Karpathy

LLM-агенты системно ошибаются одинаково. Эти четыре правила — противоядие. Они выше любых других инструкций в этом файле.

### 1.1. Думай до кода

**Не предполагай. Не прячь свою confusion. Поднимай tradeoffs.**

- Прежде чем имплементить — проговори допущения вслух. Если не уверен — **спроси**, не угадывай.
- Если задача допускает несколько интерпретаций — **назови все**, не выбирай молча.
- Если есть более простой путь — **скажи**. Push back, когда видишь, что делаем не то.
- Если что-то непонятно — **стоп**. Назови, что именно сбивает, и спроси.
- Никаких "of course!" в ответ на критику. Если пользователь прав — фикси молча. Если не прав — аргументируй.

### 1.2. Минимум кода

**Только то, что решает задачу. Ничего на будущее.**

- Никаких фич сверх того, что попросили.
- Никаких абстракций для одноразового кода.
- Никакой "гибкости" и "конфигурируемости", которую не просили.
- Никакой обработки ошибок для сценариев, которые невозможны.
- Если написал 200 строк, а можно было 50 — **перепиши**.
- **Regex до LLM**: для парсинга структурированного текста (даты, email, IDs, фиксированные форматы) — regex/parser. LLM-вызов только когда структура нерегулярная или нужно понять смысл. LLM на регулярке = переусложнение + расходы + флаки-тесты.

Тест: «Senior сказал бы, что это переусложнено?» Если да — упрости.

### 1.3. Хирургические правки

**Трогай только то, что должен. Чисти только за собой.**

При редактировании существующего кода:
- Не «улучшай» соседний код, комментарии или форматирование.
- Не рефактори то, что не сломано.
- Соблюдай существующий стиль, даже если ты бы сделал иначе.
- Заметил несвязанный мёртвый код — **скажи**, не удаляй.

Когда твои правки оставили хвосты:
- Убери импорты/переменные/функции, которые **твои** изменения сделали ненужными.
- Не удаляй мёртвый код, существовавший до тебя, если не попросили.

Тест: каждая изменённая строка должна прямо трассироваться к запросу пользователя.

### 1.4. Goal-driven исполнение

**Определи критерий успеха. Крути цикл, пока не зелёный.**

Переводи задачи в проверяемые цели:
- «Добавь валидацию» → «Напиши тесты на невалидные входы, потом сделай их зелёными».
- «Почини баг» → «Напиши тест, который воспроизводит баг, потом сделай его зелёным».
- «Зарефактори X» → «Тесты зелёные до и после».

Для многошаговых задач сразу проговори план:
```
1. [шаг] → проверка: [как пойму, что готово]
2. [шаг] → проверка: [...]
```

Сильные критерии успеха позволяют крутить цикл автономно. Слабые («сделай чтобы работало») требуют постоянных уточнений.

## 2. Какой инструмент когда

| Размер задачи | Инструмент |
|---|---|
| **Тривиальное** (≤ 10 строк, ≤ 2 файла, переименовать, добавить лог) | Edit/Write напрямую, без скиллов. |
| **Фича за одну сессию** | `superpowers:brainstorming` (если есть развилка) → `writing-plans` → `test-driven-development`. |
| **Фича с UI** | **Сначала набросок** через `frontend-design` или `gsd-sketch` (HTML-мокап) → дискуссия по нему → потом бэк и реальная имплементация. См. §3. |
| **Многофазный проект / большой модуль** | Опционально GSD: `gsd-new-project` → `gsd-spec-phase` → `gsd-plan-phase` → `gsd-execute-phase`. Создаёт `.planning/` с артефактами, которые переживают сессии. |
| **Баг** | `superpowers:systematic-debugging` (научный метод, гипотеза → проверка). |
| **Крупная фича готова** | `reflexion:critique` — мульти-перспективное ревью. Не блокирует merge, но не пропускается на чём-то большом. |

GSD по умолчанию **не запускается автоматически** в пустом репо. Если проект однофайловый — обходимся superpowers. Если идея явно крупная — пользователь скажет «погнали через GSD» или Claude сам предложит.

## 3. UI — сначала набросок, потом код

Любая фича/проект с UI начинается **не с моделей и эндпоинтов**, а с визуального наброска. Это применимо и к проекту целиком (на стадии проектирования), и к отдельной фиче с экраном.

- **На старте проекта** с UI — сначала `frontend-design` или `gsd-sketch` генерит мокапы основных экранов (HTML/Tailwind, throwaway). Обсуди их с пользователем. Только после этого — бэк, схемы, миграции.
- **На старте фичи** с UI — то же самое. Сначала набросок экрана / компонента, утверждение, потом контракт API под этот UI, потом имплементация.
- Это не «дизайн-фаза на месяц». Это **30 минут** на HTML-мокап, чтобы убедиться, что пилим то самое и в том виде, в котором ожидает пользователь.
- Без UI (CLI / API-only / библиотека) — секция не применяется.

Зачем: 90% переделок бэка случаются потому, что UI оказался не таким. Дешевле перерисовать мокап, чем мигрировать схему.

## 4. Docker — единственная среда выполнения

Если в проекте есть `docker-compose.yml`:
- Все тесты, запуски, проверки — только в контейнерах. Не запускай локально.
- Поменял код → сам пересобери/перезапусти контейнер. Не проси пользователя.
- `docker compose` предпочтительнее одиночных `docker run`.
- Тестовый стенд — `docker-compose.test.yml` с изолированной БД.

Если проекта в Docker нет — игнорируй секцию, работай как принято в стеке.

## 5. Тесты

- HTTP API: на каждый endpoint — integration-тест с реальной БД (не mock). Минимум для CRUD: `create → get → verify`.
- Тесты пишутся **вместе** с кодом, не после. См. `superpowers:test-driven-development`.
- TDD цикл: RED → GREEN → REFACTOR → COMMIT.
- Красный тест = работа не завершена.

## 6. Research first → Stop rule (три провала — стоп)

**Перед сложной имплементацией** — потрать 5 минут на разведку, это окупается:
1. `grep`/`Glob` по проекту: «такое уже делали?»
2. `claude-mem:mem-search`: «это уже решали в прошлой сессии?»
3. `mcp__plugin_context7_context7__query-docs`: «как это правильно в текущей версии библиотеки?»

Дешевле прочитать существующий паттерн за 5 минут, чем 30 минут писать своё и потом ловить грабли.

Если три раза подряд не можешь решить одну проблему:
1. Остановись. Не пробуй четвёртый раз наугад. Это и есть «manage your confusion» из Karpathy.
2. **Архитектурный вопрос** (что выбрать, как масштабировать, где узкое место) → `docs/system-design-patterns.md` (шпаргалка по Alex Xu) → дальше `context7` за деталями.
3. **Баг библиотеки** → `mcp__plugin_context7_context7__resolve-library-id` + `query-docs`. Не выдумывай API из памяти.
4. **Системный баг** → `superpowers:systematic-debugging`.
5. Сформулируй в чате: «застрял на X, прочитал Y, гипотеза Z». Спроси.

Выходи из retry-цикла после 3-го фейла и зови человека. Не пытайся четвёртый раз наугад.

## 7. Секреты, конфиги, git

- Никогда не хардкодь секреты. Всё через `.env` + `.env.example`. `.env` в `.gitignore`.
- Conventional Commits: `feat:`, `fix:`, `docs:`, `refactor:`, `test:`, `chore:`. Английский, императив.
- Коммит после каждой завершённой подзадачи, не пачкой в конце.
- Перед PR — `git pull && git rebase main`.

**Никогда без явной просьбы пользователя:**
- `git push --force`, `git reset --hard`, `git checkout .`, `git clean -f`, `git branch -D`.
- Прямой merge в `main` мимо PR.
- `--no-verify`, `--no-gpg-sign`.
- Удаление файлов, которые Claude не создавал в этой сессии.
- Изменения в `.env`, секретах, CI-конфигах.

## 8. Контекст и память

- **Большой вывод** (тесты, логи, `git log`, `find`, `grep` с тысячами строк) — через `mcp__plugin_context-mode_context-mode__ctx_batch_execute` или `ctx_execute`, **не** через Bash напрямую. Иначе вышибает контекстное окно.
- **Память между сессиями** — `claude-mem:mem-search` («это уже решали?», «как делали в прошлый раз?»). Перед тем как ломать голову — проверь.
- **Документация библиотек** — `mcp__plugin_context7_context7__resolve-library-id` → `query-docs`. Не пиши API по памяти, оно устарело.
- **`/compact` по breakpoints, не по 95%**: компакти осознанно на стыках работы — после research перед implementation, после milestone, после неудачной попытки перед новой. Не жди авто-компакта на 95% (теряются ценные детали). НЕ компакти посередине имплементации — потеряешь имена переменных и пути.

## 9. Документация после крупных изменений

После завершения крупной фичи обнови:
- `docs/changelog.md` — что изменилось, дата.
- `docs/architecture.md` — если поменялась архитектура.
- README, если поменялся способ запуска.

Мелкие правки в changelog не пишутся.

## 10. Deploy и БД

Шаблон стек-нейтрален — выбор за тобой. Шпаргалка для соло-MVP:

| Сценарий | Что брать |
|---|---|
| **Next.js / SPA + статика/SSR** | Vercel + Neon (или Supabase, если нужна auth из коробки) |
| **Бэк в Docker + БД** | Railway (бэк + Postgres в один клик) или Fly.io |
| **Edge-функции глобально** | Cloudflare Workers + D1 (плагин `cloudflare` ставится отдельно) |
| **AI-приложение с LLM** | Vercel + AI SDK + Neon. См. `vercel:ai-sdk` skill. |
| **Локально, полный стек** | `docker compose` с `postgres` / `redis` / `mongo` — пиши свой `docker-compose.yml` |
| **Только prototyping, без денег** | Vercel free + Neon free tier |

**Принципы:**
- На MVP бери managed-сервис (Vercel + Neon/Supabase + Railway). Не пиши свой Postgres в Docker и свой auth до подтверждения, что продукт нужен.
- Когда продукт подтвердился — мигрируй на self-hosted (стоимость окупится).
- Никаких хардкодов URL/ключей — всё через `.env` (см. §7).
- Каждый PR деплоится в preview-окружение (Vercel/Railway это умеют сами).

**Плагины:** `vercel` и `railway` — в required (всегда под рукой). `neon`, `supabase`, `prisma`, `auth0` — opt-in: установи через `/plugin install <name>@claude-plugins-official` только когда реально нужны.

Сначала **уточни** у пользователя. Не игнорируй молча. Не «обходи ради скорости». Это прямое следствие §1.1.
